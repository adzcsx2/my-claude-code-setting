## Personal Preferences

### Privacy

- Always redact logs; never paste secrets (API keys/tokens/passwords/JWTs)
- Review output before sharing - remove any sensitive data

### Code Style

- No emojis in code, comments, or documentation
- Prefer immutability - never mutate objects or arrays
- Many small files over few large files
- 200-400 lines typical, 800 max per file
- All files (source code, scripts, configs, and any files generated by scripts at runtime) must use **UTF-8 encoding without BOM**. Never write files with BOM (`\xEF\xBB\xBF`). Language-specific requirements: Python: use `open(..., encoding='utf-8')`, never `utf-8-sig`; Node.js: use `'utf8'` encoding; Shell: never prefix output with `\xef\xbb\xbf`; editors/tools must also be configured to write UTF-8 no-BOM.

### Git

- Conventional commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`
- Always test locally before committing
- Small, focused commits

### Testing

- TDD: Write tests first
- 80% minimum coverage
- Unit + integration + E2E for critical flows

### Knowledge Capture

- Personal debugging notes, preferences, and temporary context → auto memory
- Team/project knowledge (architecture decisions, API changes, implementation runbooks) → follow the project's existing docs structure
- If the current task already produces the relevant docs, comments, or examples, do not duplicate the same knowledge elsewhere
- If there is no obvious project doc location, ask before creating a new top-level doc

## Playwright Execution

When Playwright browser operations are needed, choose the execution method based on the task type:

**Use Playwright CLI** (`npx playwright` .js scripts via Bash) when:
- Running fixed, pre-defined test suites or automation scripts
- Batch operations that can complete in a single script execution
- Performance matters (CLI avoids MCP protocol round-trip overhead per action)
- CRITICAL: Write the full script to a temp `.js` file first, then execute it with `npx playwright`

**Use Playwright MCP** (`mcp__playwright-local__*` tools) when:
- Interactive debugging or exploratory testing step by step
- Need to inspect page state between actions to decide next steps
- One-off investigations where writing a script is overkill

**Default**: Always prefer CLI for known, repeatable test flows. Only fall back to MCP when interactivity is actually needed.
